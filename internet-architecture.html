知识点 01

基础技术架构

一个APP是怎么跑起来的？

核心概念

互联网产品的基础技术架构通常采用"三层架构"或"四层架构"模式，包含表现层、业务逻辑层和数据层。

📱

前端 / 客户端

运行在用户设备上的应用程序，负责用户界面展示和用户交互。包括Web浏览器、移动APP、桌面应用、小程序等。

🌐

网络层

负责前后端之间的数据传输，通常基于HTTP/HTTPS协议。是数据传输的"高速公路"。

⚙️

服务端 / 后端

运行在服务器上的程序，负责处理业务逻辑、数据计算、安全验证等核心功能。是产品的"真正大脑"。

🗄️

数据库

用于持久化存储和管理数据的系统，支持数据的增删改查操作。是"数据的仓库"。

生活化类比：餐厅点餐

🍽️ 餐厅类比理解

你用手机刷抖音、发微信、点外卖，整个过程就像"你去餐厅吃饭"：

技术概念 餐厅类比 通俗理解
前端 餐厅门面 + 菜单 你能看到的、能点击的界面，比如按钮、图片、文字
网络 服务员 把你的需求传给后厨，再把结果传回来
服务端 后厨 接收需求、处理逻辑、返回结果
数据库 食材仓库 存着所有数据，后厨需要什么就去拿

前端类型全景图

🍎

iOS 应用

苹果设备上的原生应用，使用 Swift/Objective-C 开发，性能最佳。

🤖

Android 应用

安卓设备上的原生应用，使用 Kotlin/Java 开发，覆盖面最广。

🌐

Web 网页

使用 HTML/CSS/JavaScript 开发，跨平台访问，无需安装。

⚡

PWA 应用

渐进式Web应用，可离线使用，接近原生体验。

💬

微信小程序

无需下载即用即走，获客成本低，开发效率高。

📱

支付宝小程序

依托支付宝生态，适合生活服务、金融场景。

🪟

Windows 应用

Windows 桌面应用，适合办公、专业软件场景。

🍎

macOS 应用

Mac 桌面应用，适合设计、开发等专业用户。

现代架构演进

单体架构
→

单体架构 初创期、用户量小 开发快、成本低

💡
产品经理需要记住
前端负责"看"，后端负责"算"，数据库负责"存"——各司其职，不要越界。
前端不能直接操作数据库——必须经过服务端，为了安全。
复杂逻辑放后端，简单展示放前端——分工明确，性能更好。

📋 本节小结

1 互联网产品采用三层架构：前端（表现层）、后端（业务逻辑层）、数据库（数据层）
2 前端是用户看到的界面，后端是产品的"大脑"，数据库是数据的"仓库"
3 架构从单体到微服务再到云原生，是业务发展的必然选择

知识点 02

请求与响应模型

用户点击按钮后发生了什么？

核心概念

请求与响应模型（Request-Response Model）是互联网通信的基础模式，基于HTTP/HTTPS协议实现。这是一种"一问一答"的同步通信模式。

📤

请求 Request

客户端（前端）向服务端发送的数据包，包含请求方法、请求地址、请求头、请求体等信息。

📥

响应 Response

服务端处理请求后返回给客户端的数据包，包含状态码、响应头、响应体等信息。

HTTP 请求结构

HTTP Request 
// 请求行
POST /api/user/login HTTP/1.1

// 请求头
Host: api.example.com
Content-Type: application/json
Authorization: Bearer xxx123
User-Agent: iOS/15.0

// 请求体
{
  "username": "xiaoming",
  "password": "123456"
}

HTTP 请求方法

🔍 GET - 获取数据

特点：参数在URL中显示，有长度限制，可以收藏为书签

场景：查看商品列表、搜索内容、查看详情

示例：GET /api/goods/list?category=手机&page=1

➕ POST - 提交数据

特点：参数在请求体中，没有长度限制，安全性较高

场景：用户登录、提交订单、上传文件

示例：POST /api/order/create

✏️ PUT - 更新数据

特点：完整更新，需要传完整数据

场景：更新用户全部信息

示例：PUT /api/user/123

🗑️ DELETE - 删除数据

特点：删除指定资源

场景：删除商品、删除评论

示例：DELETE /api/goods/123

HTTP 状态码

状态码 状态描述 产品场景 用户看到什么
200 成功 一切正常 正常显示内容
400 参数错误 用户输入了非法数据 "请输入正确的信息"
401 未登录 登录过期/未登录 跳转到登录页
403 无权限 普通用户访问管理员页面 "您没有权限访问"
404 不存在 访问了不存在的页面 "页面不存在"
500 服务器错误 服务器出bug了 "系统繁忙，请稍后"

同步 vs 异步

📞

同步请求

像打电话：必须等对方回复，期间不能做其他事。

场景：用户登录、提交订单

💬

异步请求

像发微信：发完消息就可以做其他事，等回复了再处理。

场景：加载更多、上传文件、点赞收藏

💡
产品经理需要记住
一个请求对应一个响应——没有请求就没有响应，服务器不会主动找你。
状态码告诉你发生了什么——200成功、4xx客户端问题、5xx服务器问题。
同步等结果，异步不等结果——根据业务场景选择合适的模式。

📋 本节小结

1 HTTP请求由请求行、请求头、请求体三部分组成
2 GET用于查询，POST用于创建，PUT用于更新，DELETE用于删除
3 状态码是产品经理必须了解的概念，它告诉你请求是否成功、失败原因

知识点 03

技术思维

开发是怎么思考问题的？

核心概念

技术思维是一种从实现角度看待产品设计的思维方式，核心特征包括：步骤化、严谨性、路径推理、闭环思维。

产品思维 技术思维
想的是"用户要什么" 想的是"怎么一步步实现"
关注"正常流程" 关注"正常流程 + 异常流程"
想的是"功能好不好用" 想的是"功能能不能实现、稳不稳定"
画的是"界面原型" 画的是"流程图 + 状态图"

技术思维四要素

📋

步骤化思维

将复杂功能拆解为有序的、可执行的步骤序列，每个步骤有明确的输入和输出。

🎯

边界条件思维

想清楚"什么情况下会出问题"，考虑输入或状态的极限值、临界值。

⚠️

异常处理思维

想清楚"出错了怎么办"，让用户知道发生了什么，保护用户数据。

🔄

闭环思维

事情要有始有终，不能"悬在半空"。确保每个操作都有明确的开始、过程和结束状态。

边界条件类型大全

  • 数值最小值：年龄不能小于0
  • 数值最大值：年龄不能大于150
  • 字符串为空：用户名不能为空
  • 字符串超长：评论不能超过500字
  • 特殊字符：密码不能包含emoji
  • 未登录状态：用户没有登录
  • 过期状态：登录已过期
  • 禁用状态：账号被封禁
  • 审核状态：内容正在审核中
  • 库存为0：商品卖完了
  • 余额不足：用户钱不够
  • 超出限制：购买数量超过限制
  • 重复操作：已经点过赞了
  • 时间冲突：时间段已被预约
  • 网络断开：用户没网
  • 服务器故障：服务器挂了
  • 并发冲突：两个人同时买最后一件商品
  • 兼容问题：旧版本APP不支持

案例：用户积分兑换商品

🎁 产品思维 vs 技术思维

【产品思维】

用户点击兑换 → 扣积分 → 给商品 → 完成

【技术思维】

用户点击兑换
↓
用户登录了吗？
未登录 → 提示登录
已登录 → 继续
↓
积分够吗？
不够 → 提示"积分不足"
够 → 继续
↓
商品有库存吗？
没有 → 提示"商品已兑换完"
有 → 继续
↓
兑换成功！

异常处理原则

不要让用户看到"系统错误"
❌ 不好：提示"系统错误，错误代码500" → ✅ 好：提示"服务器开小差了，请稍后重试"
告诉用户发生了什么
❌ 不好：提示"操作失败" → ✅ 好：提示"库存不足，无法购买"
告诉用户该怎么办
❌ 不好：提示"登录过期" → ✅ 好：提示"登录已过期，请重新登录" + 跳转登录页
保护用户数据
❌ 不好：扣了钱但订单没创建成功 → ✅ 好：要么都成功，要么都失败（事务回滚）
💡
产品经理需要记住
把"做什么"拆成"第一步、第二步..."——步骤化思维。
想清楚"什么情况下会出问题"——边界条件思维。
想清楚"出错了怎么办"——异常处理思维。

📋 本节小结

1 技术思维要求产品经理不仅考虑"用户要什么"，还要考虑"怎么实现"、"什么情况会出错"
2 边界条件包括：数据边界、状态边界、业务边界、环境边界
3 异常处理要遵循：友好提示、明确原因、给出解决方案、保护数据

知识点 04

技术边界

什么能做，什么做不了？

核心概念

技术边界是指在当前技术水平和资源条件下，产品需求能够实现的范围和限制。不是你想设计什么功能，开发就能做出来。

🔧

能力边界

技术上能否实现？涉及硬件、软件、算法、数据、合规等方面。

💰

成本边界

投入是否可接受？涉及开发成本、服务器成本、运营成本。

⏰

时间边界

周期是否可接受？涉及功能复杂度、团队情况、技术依赖。

✅

质量边界

效果能否达标？涉及性能、稳定性、安全性、兼容性。

能力边界分类

边界类型 关键问题 典型案例
硬件能力 手机硬件支持吗？ GPS定位、摄像头扫码、NFC支付
软件能力 操作系统允许吗？ iOS后台定位限制、Android权限管理
算法能力 技术成熟可靠吗？ 人脸识别成熟、情绪识别不成熟
数据能力 有足够数据吗？ 推荐系统需要用户行为数据
合规能力 法律允许吗？ 人脸采集需要用户明确授权

成本评估参考

功能类型 开发成本 服务器成本 典型场景
简单功能 2-5人天 极低 点赞、收藏、分享
中等功能 1-2人周 低 评论系统、搜索功能
复杂功能 1-2人月 中 支付系统、消息推送
高级功能 3人月+ 高 推荐系统、AI能力

案例：人脸识别功能分析

✅ 成熟能力

1:1人脸比对准确率99%+、活体检测、多角度识别

⚠️ 有限能力

光线不足时识别率下降、戴口罩/墨镜识别困难

❌ 不成熟

1:N大规模人脸搜索、跨年龄识别、情绪识别

方案 成本 周期 适合场景
第三方API 0.01-0.1元/次 1-2周 中小团队、快速验证
自建方案 算法工程师40-80万/年 3-6个月 日调用>100万次
⚠️
合规要求
• 《个人信息保护法》：明确告知、单独同意
• 《网络安全法》：数据加密存储、传输
• 必须提供替代登录方式（密码、验证码）
• 用户可随时撤回授权、删除人脸数据

技术决策框架：FEASIBLE模型

Function 功能明确？
Environment 环境支持？
Ability 能力足够？
Schedule 时间充足？
Investment 投入可接受？
Benefit 收益如何？
Law 合规吗？
Experience 体验达标？

总分 ≥32分：高可行性 | 24-31分：中可行性 | <24分：低可行性

技术边界自检清单

能力边界
这个功能技术上能实现吗？实现到什么程度？有没有技术难点？
成本边界
开发成本是多少？服务器成本是多少？投入产出比如何？
时间边界
开发周期是多久？这个时间能接受吗？能否分期上线？
质量边界
性能要求是什么？稳定性要求是什么？安全性要求是什么？
💡
产品经理需要记住
能做 ≠ 能做好——问清楚能做到什么程度，关注准确率、性能等指标。
竞品能做 ≠ 你也能做——考虑自己的资源、能力和数据积累。
技术问题不一定能用钱解决——技术积累和数据积累需要时间。

📋 本节小结

1 技术边界包括：能力边界、成本边界、时间边界、质量边界
2 使用FEASIBLE模型系统性评估功能可行性
3 遇到技术边界限制时，可以：简化方案、替代方案、MVP方案、妥协方案
